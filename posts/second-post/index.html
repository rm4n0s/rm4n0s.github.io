What is Odin and who is it for

  • Odin is a new programming language that is

    • 60% Golang,
    • 20% feature that you wished Golang had
    • 10% array programming
    • 10% low level, FFI, memory management

  • People think that it is only for game development just because the creator of Odin, Ginger Bill, included 3D libraries in the language, but I disagree. Odin is also for backend and devops, because Ginger Bill copied 70% of Go’s standard library to the language, which means it has all the building blocks for us to copy the rest of Go’s libraries like chinese manufacturers.

  • and I know that some of Golang’s developers have moved already to Zig, C3 and Hare, but those languages do not have array programming which can be used on backend development for:

    • finance and trading
    • AI
    • cryptography
    • multimedia
    • server side anti-cheat (what every linux gamer wish for)

Is it ready?

Yes, it is, the language will not change and it will stay stable until you die. However, the compiler, toolchain, and core library are still in development and always improved

If it is ready, prove it!

It is used in production from JangaFX and ChiAha™

What are the characteristics of the language?

The language has 31 keywords but each keyword may have a #directive or @(attribute). It may seem too much for someone that cannot remember more than 26 keywords like me, but all the keywords, directives and attributes connect so seamlesly together like reading english. For example, the switch-case that works like in Golang requires from the user to specify every case, unless you put the #partial directive in front of it.

It will really make sense to you how easily you can understand the code without knowing the language. Just stop for 10 minutes and read this code that has almost everything https://github.com/odin-lang/Odin/blob/master/examples/demo/demo.odin

Yes, it does not have documentation (no, the “overview” is not a real documentation), but that is the beauty of it. You learn to code in Odin by reading other’s people code. For example, even though It has struct tags that works the same way as in Golang, I didn’t know which package uses them, because there are no examples, so I had to open each package and read the code. So far only “core:encoding/json” package use them.

It does not have macros, or comptime, or constexpr and it is better that way because from personal experience every language that has them makes the developer to waste 90% of his time googling for every library that uses them. Trust me, stick with Odin’s pre-defined directives and attributes.

Yes, it has generics and they are more simpler than Golang’s generics.

What are the trade offs?

First, Odin has manual memory management and for that reason it does not have Closures, Composition and Goroutines (yes you can live without them). Secondly, it does not have libraries, for that reason, I am trying to convince you to learn it and copy your libraries from Go to Odin. Moreover, it does not have documentations, so you will take more time to feel comfortable writting with memory allocation, threads, interfaces and errors. For that reason, the rest of the article will explain to you about these four pain points so you can start copying your Go library to Odin in no time.
