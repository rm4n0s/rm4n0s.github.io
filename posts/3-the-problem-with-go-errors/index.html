Introduction

  • In the previous article I compared Golang’s and Odin’s error handling, and some golang developers told me that it is possible to do the same thing with errors.Is(), errors.As(), fmt.Errorf() or other library.

At that point, I realised that developers didn’t understand the problem or even worse they don’t know their rights. They don’t know how their programming language restricts their developers rights

This article is for all these people who believe that their programming language can challenge Odin’s superior error handling. For that reason, I challenge you to the “Error Handling Challenge!”

  • Errors have changed over the years

    • C
      • return number for error
      • pointer error as parameter (for giving more information)
    • C++/Python/Java/JS
      • exceptions
    • Go/Zig/Rust
      • return specific error type
    • Odin
      • return any type

  • even though the errors changed over the year, what developers want never changed

Developer’s rights

  • know what errors a function returns without reading the function’s code
    • through the description of the function
    • from the type of the error it returns
    • or the IDE tells you
  • know the list of functions called that produced each and any error in production
  • parse error’s the list of functions called to produce a message for the user

To find which programming language supports all the developer’s rights, then we have to do a challenge

The challenge

This challenge is for all developers and they can use any programming language, but the example is in Golang.

func f1() error {
	n := rand.IntN(9) + 1
	if n%2 == 0 {
		return io.ErrNoProgress
	}
	return io.ErrUnexpectedEOF
}

func f2() error {
	return f1()
}

func f3() error {
	return f1()
}

func f4() error {
	n := rand.IntN(9) + 1
	if n%2 == 0 {
		return f2()
	}
	return f3()
}

func main() {
	err := f4()

	// print three different messages based on the execution path of the function:
	// - for f4()->f2()->f1()->io.ErrUnexpectedEOF print "Aand it's gone"
	// - for f4()->f3()->f1()-io.ErrNoProgress print "The money in your account didn't do well"
	// - for the rest of the cases print "This line is for bank members only"


    // also print any type of stack trace for err
    fmt.Println("Print any type of stack trace that shows the path of execution for ", err)
}

To do the challenge in your own programming language, please don’t change the logic or the number of functions or the messages that you have to print, but you can change the type of errors and use any method to track the path of function calls.
